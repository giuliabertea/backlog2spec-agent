## JSON Schema

{
  "workItemId": <integer>,
  "title": "<string>",
  "missingAcceptanceCriteria": ["<string>", ...],
  "edgeCases": ["<string>", ...],
  "constraints": ["<string>", ...],
  "affectedComponents": ["<string>", ...],
  "ambiguities": ["<string>", ...]
}

Rules:
- Do NOT guess or invent information not supported by the work item
- If a list cannot be determined from the work item, leave it as an empty array []
- Every string in missingAcceptanceCriteria should describe a testable condition that is NOT covered by the existing acceptance criteria
- edgeCases should be boundary conditions or failure scenarios the developer should handle
- constraints should be non-functional requirements implied by the context (performance, security, compatibility)
- affectedComponents should use real class or service names from the Project Context or Wiki Context where possible
- ambiguities should be questions that must be answered before implementation can begin
